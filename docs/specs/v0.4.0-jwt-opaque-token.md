# v0.4.0 — JWT + Opaque Token 技术规格

> 版本：v0.4.0
> 状态：已冻结
> 前置依赖：v0.3.2（API Key + scope + 配额）
> 关联 ADR：[ADR-007](../adr/007-hybrid-token-model.md)

---

## 1. 目标

在现有 API Key 基础上新增 JWT 和 Opaque Token 两种凭证类型。`/introspect` 自动识别三种 Token 格式。为 v1.0 OAuth2 铺设 JWT 基础设施。

## 3. Token 类型检测

`/introspect` 根据 token 前缀自动选择验证路径：

| 前缀 | Token 类型 | 验证路径 |
|------|:--:|------|
| `pk_live_` | API Key | sha256 → Redis → PostgreSQL |
| `eyJ` | JWT | RS256 验签 → exp 检查 → Redis 吊销集 |
| `ot_` | Opaque | sha256 → Redis → PostgreSQL |
| 其他 | — | 返回 `{ active: false }` |

`token_type_hint` 参数可覆盖自动检测（可选）。

## 4. TokenVerifier — 统一 dispatch 架构

**设计原则**：HTTP handler 不感知 token 类型。`TokenVerifier` 是统一入口，内部根据前缀 dispatch 到三个子 verifier。

```rust
pub struct TokenVerifier {
    api_key_verifier: ApiKeyVerifier,
    jwt_verifier: JwtVerifier,
    opaque_verifier: OpaqueVerifier,
}

impl TokenVerifier {
    pub async fn verify(&self, token: &str) -> IntrospectResponse {
        if token.starts_with("pk_live_") {
            self.api_key_verifier.verify(token).await
        } else if token.starts_with("eyJ") {
            self.jwt_verifier.verify(token).await
        } else if token.starts_with("ot_") {
            self.opaque_verifier.verify(token).await
        } else {
            IntrospectResponse::inactive()
        }
    }
}
```

- `ApiKeyVerifier` — 已有，不变
- `JwtVerifier` — 新增，本地验签 + Redis 吊销集检查
- `OpaqueVerifier` — 新增，与 ApiKeyVerifier 共用 sha256→Redis→PG 路径，但使用 `ot_` 前缀和独立 TTL

## 5. JWT 设计

### 5.1 密钥管理

- RS256 非对称签名（私钥签发，公钥验签）
- 密钥对从环境变量加载或自动生成：
  - `JWT_PRIVATE_KEY_PEM` — 私钥（PEM 格式，仅 Aspectus 持有）
  - `JWT_PUBLIC_KEY_PEM` — 公钥（PEM 格式，分发给各项目）
- 未配置时自动生成临时密钥对（开发环境）
- Key rotation 预留：支持多个 key（通过 `kid` 区分），当前 v0.4 只用一个

### 5.2 Public Key 分发

```
GET /.well-known/jwks.json

Response 200:
{
  "keys": [
    {
      "kty": "RSA",
      "kid": "aspectus-v0.4-001",
      "n": "...",
      "e": "AQAB"
    }
  ]
}
```

各项目启动时拉取此端点，缓存 public key，用于本地验签。

### 5.3 JWT Claims

```json
{
  "sub": "sa_abc123def456789",
  "tenant_id": "org_acme",
  "scope": "pandaria:session:create pandaria:session:read",
  "client_id": "pandaria",
  "iat": 1717000000,
  "exp": 1717000900,
  "jti": "unique-jti-abc123",
  "iss": "https://aspectus",
  "aud": "pandaria"
}
```

| Claim | 说明 |
|-------|------|
| `sub` | Service Account ID (来自 `api_key.service_account_id`) |
| `tenant_id` | 租户 ID |
| `scope` | 空格分隔的 scope 列表 |
| `client_id` | Project 名称 |
| `iat` | 签发时间 |
| `exp` | 过期时间（5-15 分钟，可配置 `JWT_TTL_SECONDS`，默认 900） |
| `jti` | 唯一 Token ID，21 字符 hex（用于吊销） |
| `iss` | 签发者 |
| `aud` | 目标 Project（= `client_id`），限制 JWT 的接收方 |

### 5.4 `/introspect` 中 JWT 验证流程

```
1. 检查前缀：token.starts_with("eyJ")
2. RS256 验签（public key）
   → 验签失败 → { active: false }
3. 检查 exp（过期时间）
   → 已过期 → { active: false }
4. 检查 jti 是否在 Redis 吊销集合中
   → 在集合中 → { active: false }
5. 从 claims 提取字段构建 IntrospectResponse
   → { active: true, token_format: "jwt", tenant_id, user_id: sub, identity_type: "service_account", ... }
```

**关键差异 vs API Key**：JWT 验证**不查 PostgreSQL**。所有信息在 claims 中自包含。只有吊销检查需要 Redis。

### 5.5 JWT 签发

```
POST /token
Authorization: Bearer {service_token}
Content-Type: application/json

{
  "grant_type": "client_credentials",
  "client_id": "sa_abc123",         // Service Account ID
  "client_secret": "pk_live_...",   // 该 SA 的任意有效 API Key
  "token_format": "jwt"             // 可选，默认 "jwt"
}

Response 200:
{
  "access_token": "eyJhbGciOiJSUzI1NiIs...",
  "token_format": "jwt",
  "expires_in": 900,
  "token_type": "Bearer"
}
```

**签发逻辑**：
1. 验证 Service Token（调用方身份）
2. 验证 `client_secret`：通过 `ApiKeyVerifier.verify(client_secret)` 检查 API Key 有效性
3. 从 API Key 获取 `tenant_id`、`scopes`、`project`
4. 构建 JWT claims
5. RS256 签名 → 返回 JWT

### 5.6 JWT 吊销

```
POST /token/revoke
Authorization: Bearer {service_token}
Content-Type: application/json

{ "token": "eyJ..." }

Response 204
```

**吊销逻辑**：
1. 解码 JWT（不验签）提取 `jti` 和 `exp`
2. 执行 `SETEX jwt_revoked:{jti} {exp - now} 1`（per-key 方式，JWT 过期后自动清除）
3. 返回 204

已吊销的 JWT 再次吊销 → 幂等返回 204。

## 6. Opaque Token 设计

### 6.1 格式与存储

与 API Key 相同格式（随机 hex 字符串），但存储在独立的 `subject_tokens` 表（v1.0 激活；v0.4 用 `api_keys` 表加 `token_type` 列区分）。

**v0.4 简化**：Opaque Token 也存 `api_keys` 表，新增 `token_type` 列：

```sql
ALTER TABLE api_keys ADD COLUMN token_type VARCHAR(16) NOT NULL DEFAULT 'api_key';
-- token_type: 'api_key' | 'opaque' | (v1.0: 'jwt' 存储 jti)
```

### 6.2 `/introspect` 中 Opaque 验证

与 API Key 完全相同的路径：sha256 → Redis → PostgreSQL。区别仅在于：
- Opaque token 使用 `ot_` 前缀（如 `ot_aBcDeFgH...`），与 `pk_live_` 区分
- 响应中 `token_format: "opaque"`
- 默认 TTL 较短（可配置 `OPAQUE_TOKEN_TTL_SECONDS`，默认 3600s）

### 6.3 Opaque Token 签发

```
POST /token
{
  "grant_type": "client_credentials",
  "client_id": "sa_abc123",
  "client_secret": "pk_live_...",
  "token_format": "opaque"         // 指定格式
}

Response 200:
{
  "access_token": "ot_abc123...",
  "token_format": "opaque",
  "expires_in": 3600,
  "token_type": "Bearer"
}
```

### 6.4 Opaque Token 吊销

通过 `POST /token/revoke`（统一吊销端点，自动识别 token 类型），与 API Key / JWT 相同。`DELETE /api-keys/{id}` 仅用于 API Key，不处理 Opaque Token。

## 7. `/introspect` 响应扩展

v0.4 新增 `token_format` 字段。`IntrospectResponse` struct 增加：

```rust
#[serde(skip_serializing_if = "Option::is_none")]
pub token_format: Option<String>,  // "api_key" | "jwt" | "opaque"
```

示例响应：

```json
// API Key
{ "active": true, "token_format": "api_key", "tenant_id": "...", ... }

// JWT
{ "active": true, "token_format": "jwt", "tenant_id": "...", ... }

// Opaque Token
{ "active": true, "token_format": "opaque", "tenant_id": "...", ... }

// 无效
{ "active": false }
```

## 8. Migration

```sql
-- Migration: 20260531000006_add_token_type.sql
-- 描述：v0.4.0 — api_keys 表支持多种 token 类型

ALTER TABLE api_keys ADD COLUMN token_type VARCHAR(16) NOT NULL DEFAULT 'api_key';

-- token_type values: 'api_key' | 'opaque'
-- v1.0 可能添加 'jwt'（用于存储 jti 索引）

COMMENT ON COLUMN api_keys.token_type IS 'Token type: api_key, opaque';
```

> **v0.4 简化**：Opaque Token 与 API Key 共用 `api_keys` 表，通过 `token_type` 列区分。语义上「api_keys」表名不清，v1.0 考虑重构为 `credentials` 表。

## 10. 新增 API 汇总

| 方法 | 路径 | 认证 | 说明 |
|------|------|:--:|------|
| POST | `/token` | Service Token | 签发 JWT 或 Opaque Token |
| POST | `/token/revoke` | Service Token | 吊销 JWT 或 Opaque Token |
| GET | `/.well-known/jwks.json` | 无 | Public key 分发 |
