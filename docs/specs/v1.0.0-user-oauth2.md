# v1.0.0 — User + OAuth2 + Role 技术规格

> 版本：v1.0.0
> 状态：设计中
> 前置依赖：v0.4.0（JWT + Opaque Token）
> 关联 ADR：[ADR-004](../adr/004-user-vs-service-account-role-type.md)、[ADR-005](../adr/005-role-global-definition.md)、[ADR-013](../adr/013-emerald-entity-id-mapping.md)

## 1. 目标

人类用户可通过 OAuth2 Authorization Code 登录。Role 管理上线。API 进入长期稳定（semver 保护）。v1.0.0 是对外承诺「不再 breaking change」的里程碑。

## 3. User 模型

### 3.1 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/users` | 管理员创建用户（含初始密码） |
| GET | `/users/{id}` | 查询用户（不含 password_hash） |
| GET | `/users?tenant_id=` | 列出 tenant 下用户 |
| PUT | `/users/{id}/disable` | 禁用用户 |

### 3.2 密码

- argon2id 哈希（`aspectus-auth` 新增 `PasswordHasher`）
- 密码强度：最少 8 字符
- `POST /users` 创建时设置初始密码，首次登录后可修改（v1.0 不做自助修改）

---

## 4. OAuth2 Authorization Code Flow

### 4.1 端点

| 方法 | 路径 | 认证 | 说明 |
|------|------|:--:|------|
| GET | `/authorize` | 无 | 授权端点（返回重定向到 login 页面） |
| POST | `/token` | 无（client_secret） | 原有 + 新增 `grant_type=authorization_code` |

### 4.2 Flow

```
1. 用户访问 Pandaria → Pandaria 重定向到 Aspectus /authorize
   ?response_type=code&client_id=pandaria&redirect_uri=...&scope=...&state=...

2. Aspectus /authorize 验证 client_id →
   重定向到 login 页面（各项目自建 UI，Aspectus 返回 API）

3. 用户输入 email/password → Aspectus 验证 →
   生成 authorization code → 重定向回 redirect_uri?code=xxx

4. Pandaria 用 code 调 POST /token
   grant_type=authorization_code&code=xxx&client_id=pandaria&client_secret=...

5. Aspectus 验证 code → 返回 JWT access_token + opaque refresh_token
```

### 4.3 Token 类型复用

- **Access Token** = JWT (v0.4.0 JwtSigner)，TTL = 15 min
- **Refresh Token** = Opaque (v0.4.0 create_opaque)，TTL = 30 days
- `POST /token` 同时支持 `authorization_code` 和 `refresh_token` grant_type

## 5. Role 管理

### 5.1 Role 种子数据

| Role | type | scopes |
|------|------|--------|
| `tenant-admin` | both | `pandaria:*, tavern:*, constell:*, tokencamp:*, heirloom:*`（全 project 管理） |
| `agent-developer` | user | `pandaria:session:*, pandaria:agent:*, tavern:workflow:run, tavern:workflow:read, constell:agent:publish` |
| `agent-operator` | user | `pandaria:session:read, pandaria:agent:execute, tavern:workflow:run` |
| `ci-deployer` | service_account | `pandaria:session:create, tavern:workflow:deploy` |
| `readonly` | both | 所有 project 的 `*:read` scope |

### 5.2 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/roles` | 列出所有 Role |
| POST | `/users/{id}/roles` | 为用户分配 Role（body: `{ "role_id": "..." }`） |
| DELETE | `/users/{id}/roles/{role_id}` | 移除用户 Role |

### 5.3 role_type 约束激活

```sql
ALTER TABLE users_roles
    ADD CONSTRAINT users_roles__role_type
    CHECK (check_role_type(role_id, ARRAY['user', 'both']::role_type[]));
```

DB 层强制：User 只能分配 `type=user` 或 `type=both` 的 Role。SA 同理。

### 5.4 `/introspect` 对 User 的响应

```json
{
  "active": true,
  "token_format": "jwt",
  "identity_type": "user",
  "tenant_id": "...",
  "user_id": "user_abc123",
  "client_id": "pandaria",
  "scope": "展开后的 scope 列表",
  "exp": 1717000900
}
```

User 的有效 scope = 所有分配 Role 的 scope 并集（展开后去重）。

## 6. Emerald entity_id 迁移

**不在 Aspectus 代码中实现**。Aspectus 只提供正确格式的 `tenant_id` + `user_id`。Pandaria 的 `EmeraldMemoryStore` adapter 侧负责将 `entity_id` 从 `tenant_id` 改为 `tenant_id:user_id`。

迁移文档在 ADR-013 中已定义：
- Phase 1: entity_id = tenant_id
- Phase 2: entity_id = tenant_id:user_id（新 session 用新格式，旧数据不迁移）

## 7. API 稳定性承诺

v1.0.0 起：
- `/introspect` 响应格式进入 semver 严格管理
- 管理 API 端点签名不再 breaking change（只增不删不改）
- 新增字段使用 `Option` + `skip_serializing_if` 保证向后兼容
- v2.0.0 之前不删除任何已发布的字段
