# v0.5.0 — User + Role 管理 技术规格

> 版本：v0.5.0
> 状态：已冻结
> 前置依赖：v0.4.0（JWT + Opaque Token）
> 关联 ADR：[ADR-004](../adr/004-user-vs-service-account-role-type.md)、[ADR-005](../adr/005-role-global-definition.md)、[ADR-013](../adr/013-emerald-entity-id-mapping.md)

## 1. 目标

人类用户可被管理员创建和管理。Role 分配上线——User 通过 Role 获得 scope。`/introspect` 对 User token 返回展开后的 scope。OAuth2 推迟到 v0.6.0。

## 3. User 管理

### 3.1 端点

| 方法 | 路径 | 认证 | 说明 |
|------|------|:--:|------|
| POST | `/users` | Service Token | 管理员创建用户 |
| GET | `/users/{id}` | Service Token | 查询用户（不含 password_hash） |
| GET | `/users?tenant_id=` | Service Token | 列出 tenant 下用户 |
| PUT | `/users/{id}/disable` | Service Token | 禁用用户（不删除） |

### 3.2 请求/响应

```
POST /users
{
  "tenant_id": "org_...",
  "email": "alice@acme.com",
  "password": "min-8-chars",
  "display_name": "Alice"
}

Response 201:
{
  "id": "user_abc123",
  "tenant_id": "org_...",
  "email": "alice@acme.com",
  "display_name": "Alice",
  "is_suspended": false,
  "created_at": "2026-06-01T00:00:00Z"
}
// 不返回 password_hash
```

**校验**：
- `email` 格式 + 同 tenant 内唯一
- `password` ≥ 8 字符（不做复杂度要求，v0.5 MVP）
- `tenant_id` 必须存在

### 3.3 密码存储

- argon2id（`PasswordHasher` 新增），参数：`salt_len=16`, `hash_len=32`
- 密码哈希存在 `users.password_hash`，使用 `password_encryption_method = 'Argon2id'`

### 3.4 User 认证（v0.5 简化）

v0.5 **不做 OAuth2**。User 如何获得 token？

- 管理员为用户创建 API Key（通过 `POST /api-keys`，service_account_id 改为支持 `user_id`）
- 用户用 API Key 调 `/introspect` → `identity_type: "user"`
- User 的 scope = 分配 Role 的展开结果

> 这意味着 v0.5 `api_keys` 表需要支持 `user_id` 列（当前只有 `service_account_id`）。

## 4. `/introspect` 对 User 的响应

```json
{
  "active": true,
  "token_format": "api_key",
  "identity_type": "user",
  "tenant_id": "org_acme",
  "user_id": "user_abc123",
  "client_id": "pandaria",
  "scope": "pandaria:session:create pandaria:agent:execute tavern:workflow:run",
  "token_type": "Bearer"
}
```

**关键变化**：
- `identity_type` 从固定 `"service_account"` 变为动态（User 或 SA）
- `scope` 不再从 API Key 直接取，而是**从 Role 展开**（v0.5 新增逻辑）
- Pandaria 拿到 `tenant_id` + `user_id` 后自行拼接 `entity_id = "tenant_id:user_id"`

## 5. Role 管理

### 5.1 Role 种子数据

```sql
INSERT INTO roles (id, name, description, type, is_default) VALUES
('role_tenant_admin', 'tenant-admin', 'Full tenant management', 'both', false),
('role_agent_dev', 'agent-developer', 'Agent development access', 'user', true),
('role_agent_op', 'agent-operator', 'Agent operation access', 'user', false),
('role_ci_deployer', 'ci-deployer', 'CI/CD deployment', 'service_account', false);
```

### 5.2 roles_scopes 映射

| Role | Scope name |
|------|-----------|
| `tenant-admin` | 所有 19 个 scope（全部 project） |
| `agent-developer` | sc_pa_session_create, sc_pa_session_read, sc_pa_session_delete, sc_pa_session_manage, sc_pa_agent_execute, sc_pa_agent_manage, sc_tv_workflow_run, sc_tv_workflow_read, sc_co_agent_publish |
| `agent-operator` | sc_pa_session_read, sc_pa_agent_execute, sc_tv_workflow_run |
| `ci-deployer` | sc_pa_session_create, sc_tv_workflow_deploy |

> 不使用通配符 scope——显式列出每个具体 scope ID。v0.5 roles_scopes 全部使用 `scopes` 表中已存在的具体 scope。

### 5.3 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/roles` | 列出所有 Role |
| POST | `/users/{id}/roles` | 分配 Role（body: `{ "role_id": "..." }`） |
| DELETE | `/users/{id}/roles/{role_id}` | 移除 Role |

### 5.4 role_type 约束

```sql
ALTER TABLE users_roles
    ADD CONSTRAINT users_roles__role_type
    CHECK (check_role_type(role_id, ARRAY['user', 'both']::role_type[]));
```

## 6. Scope 展开逻辑 (`ScopeExpander`)

```
输入: user_id
1. SELECT role_id FROM users_roles WHERE user_id = $1
2. SELECT scope_id FROM roles_scopes WHERE role_id IN (...)
3. SELECT name FROM scopes WHERE id IN (...)
4. 去重 → 空格分隔字符串
```

> v0.5 不做通配符展开（roles_scopes 使用具体 scope ID）。不做缓存（4 role × 19 scope = 轻量）。v0.6+ 加 Redis TTL=60s。

## 7. `api_keys` 表扩展

v0.5 API Key 需要同时支持 User 和 SA：

```sql
ALTER TABLE api_keys ADD COLUMN user_id VARCHAR(21) REFERENCES users(id);
ALTER TABLE api_keys ALTER COLUMN service_account_id DROP NOT NULL;
-- 一个 API Key 要么属于 user 要么属于 service_account，不能同时
ALTER TABLE api_keys ADD CONSTRAINT api_keys__one_owner
    CHECK ( (user_id IS NOT NULL AND service_account_id IS NULL) OR
            (user_id IS NULL AND service_account_id IS NOT NULL) );
```

`ApiKeyCreator.create` 改为接受 `owner: IdentityOwner` enum（User 或 SA）。\n\n### 7.2 `POST /api-keys` 请求体变更\n\n```\nPOST /api-keys\n{\n  \"owner_type\": \"user\",            // \"user\" | \"service_account\"\n  \"owner_id\": \"user_abc123\",       // User ID or SA ID\n  \"project\": \"pandaria\",\n  \"scopes\": []                      // v0.5: User 的 API Key 忽略此字段，scope 从 Role 展开\n}\n```\n\nHandler 根据 `owner_type`:\n- `\"user\"` → 查 `users` 获取 `tenant_id`，API Key 的 scope 来自 Role 展开\n- `\"service_account\"` → 查 `service_accounts` 获取 `tenant_id`（现有行为不变）
