## v0.3.1 — 补齐与清理

> 目标：消除技术债务，补齐遗漏端点。不改 API 契约。

### 功能清单

- [ ] **移除所有 compiler warnings**
  - `PgServiceTokenStore` unused import in lib.rs
  - `token_type_hint` dead_code warning
  - `aspectus-client` dead_code warnings（stub fields）

- [ ] **补齐缺失的 CRUD 端点**
  - `DELETE /service-accounts/{id}` — 删除 SA（级联吊销所有关联 API Key）
  - `GET /service-accounts/{id}` — 查询单个 SA
  - `GET /api-keys/{id}` — 查询单个 API Key（不含 hash/原文）

- [ ] **补齐 Store trait 方法**
  - `TenantStore::delete` / `TenantStore::list`
  - `ServiceAccountStore::delete`
  - `ApiKeyStore::delete`（级联处理）

- [ ] **请求校验增强**
  - POST /tenants: name 非空 + 长度校验
  - POST /service-accounts: tenant_id 存在性校验
  - POST /api-keys: service_account 存在性 + project 合法性校验（已有部分）
  - 统一返回 RFC 7807 ValidationError（含 field 级别错误）

- [ ] **错误处理一致性**
  - 所有 handler 统一用 `ProblemDetails::from(err)` 模式
  - 500 错误不暴露内部细节，改为 trace_id 关联日志

- [ ] **日志改进**
  - /introspect 采样日志（每 1000 次记录 1 次延迟分布）
  - 管理 API 操作日志（actor、action、result）

- [ ] **健康检查增强**
  - `/health` 改为返回 JSON：`{"status":"ok","db":"ok","redis":"ok"}`
  - 检查 DB 和 Redis 连通性

### 验收

```bash
cargo clippy --all-targets --all-features  # 0 warnings
cargo test --workspace                      # 全部通过
curl :3100/health | jq .                    # {"status":"ok","db":"ok","redis":"ok"}
```

## v0.3.2 — 性能优化

> 目标：提升 /introspect p95 延迟和吞吐量。不改变业务逻辑。

### 功能清单

- [ ] **Redis 批量预热**
  - 启动时从 PostgreSQL 加载活跃 API Key 到 Redis（可选，通过配置开关）
  - 避免冷启动时大量 DB 查询

- [ ] **连接池参数调优**
  - PostgreSQL: 根据负载调整 `max_connections`（默认 20）
  - Redis: ConnectionManager pool size 调整
  - 添加连接池 metrics（active/idle connections）

- [ ] **DB 查询优化**
  - `api_keys` 表添加 `(tenant_id, project)` 复合索引（管理 API list 查询）
  - `service_accounts` 表 `(tenant_id, created_at)` 索引
  - 审计日志查询优化（按时间范围 + action 组合）
  - EXPLAIN ANALYZE 验证关键查询

- [ ] **响应序列化优化**
  - `/introspect` 响应预分配容量（`serde_json::to_writer` 而非 `to_string`）
  - 减少 `clone()` 调用路径
  - `quota` JSON 字段懒解析

- [ ] **并发测试**
  - `wrk` 或 `oha` 压测 `/introspect`（1000 QPS, 10s）
  - 验证 p95 < 5ms（Redis hit）
  - 验证 p95 < 10ms（Redis miss → DB）
  - 记录性能基线

- [ ] **内存优化**
  - `Arc<dyn Store>` 去重（多个 handler 共享同一 Store 实例）
  - `AppState` 字段精简

### 验收

```bash
# 压测
oha -n 10000 -c 50 -m POST \
  -H "Authorization: Bearer $SVC" \
  -d "token=$KEY" \
  http://localhost:3100/introspect

# → p95 < 5ms (Redis hit)
# → p99 < 10ms
```

## v0.4.0 — JWT + Opaque Token

> 目标：扩展 Token 类型。Service Account 可选使用 JWT 或 Opaque Token（替代 API Key）。API Key 仍然保留。

### 背景（ADR-007）

当前 v0.3 只支持 API Key（长期随机字符串）。v0.4 补充两种新 Token 类型：

| Token 类型 | 格式 | 验证路径 | 吊销 | 适用场景 |
|-----------|------|---------|------|---------|
| API Key (已有) | 随机 hex | sha256 → Redis → PG | ✅ | 长期 Agent SDK |
| **JWT (新增)** | RS256 签名 | 本地验签 + Redis 吊销列表 | ✅ | 高频服务间调用 |
| **Opaque (新增)** | 随机 hex | sha256 → Redis → PG | ✅ | 需吊销能力的短期 token |

### 功能清单

#### 4.1 JWT 基础设施

- [ ] JWT 签名密钥管理
  - RS256 密钥对生成（或从环境变量加载）
  - Public key 分发端点：`GET /.well-known/jwks.json`
  - Key rotation 支持（多 key 并存，kid 区分）

- [ ] JWT Claims 定义
  ```json
  {
    "sub": "sa_abc123",        // ServiceAccount ID
    "tenant_id": "org_acme",
    "scope": "pandaria:session:create",
    "client_id": "pandaria",
    "iat": 1717000000,
    "exp": 1717000900,         // 短有效期 (5-15 min)
    "jti": "unique-token-id"
  }
  ```

- [ ] JWT 签发端点
  - `POST /token` — 接受 `grant_type=client_credentials` + `client_id` + `client_secret`
  - 返回 `{ access_token (JWT), expires_in, token_type: "Bearer" }`
  - 签发时验证 Service Account 的 client_secret（即 API Key）

- [ ] JWT 验证（在 `/introspect` 中）
  - 检查 token 前缀区分类型：`eyJ...` → JWT
  - 本地验签（RS256 public key）→ 微秒级
  - 检查 jti 是否在 Redis 吊销集合中
  - 不查 PostgreSQL（self-contained）

- [ ] JWT 吊销
  - `POST /token/revoke` — 将 jti 加入 Redis SET（TTL = JWT 原始过期时间）
  - `/introspect` 验证时检查 Redis SET

#### 4.2 Opaque Token 基础设施

- [ ] Opaque Token 签发
  - `POST /token` 的 `grant_type=client_credentials` 也支持返回 opaque token
  - 通过 `token_format` 参数选择（默认 JWT）

- [ ] Opaque Token 验证（在 `/introspect` 中）
  - 与 API Key 共用验证路径：sha256 → Redis → PostgreSQL
  - 区别：Opaque 存储在 `subject_tokens` 表（v1.0 激活；v0.4 临时用 `api_keys` 表，加 `token_type` 列区分）

- [ ] Opaque Token 吊销
  - 与 API Key 相同：`UPDATE revoked_at = now()` + 删 Redis 缓存

#### 4.3 `/introspect` 多 Token 类型支持

- [ ] Token 类型自动检测
  - `pk_live_*` → API Key
  - `eyJ*` → JWT
  - 其他 → Opaque Token
  - `token_type_hint` 可选覆盖

- [ ] 响应中新增 `token_format` 字段
  ```json
  {
    "active": true,
    "token_format": "jwt",    // "api_key" | "jwt" | "opaque"
    ...
  }
  ```

#### 4.4 Token 管理 API

- [ ] `POST /token` — 签发 Token（client_credentials flow）
- [ ] `POST /token/revoke` — 吊销 Token
- [ ] `GET /token/info` — 查询 Token 信息（不含原文）

### 验收

```bash
# JWT flow
TOKEN=$(curl -s -X POST :3100/token \
  -H "Content-Type: application/json" \
  -d '{"grant_type":"client_credentials","client_id":"sa_...","client_secret":"pk_live_..."}' \
  | jq -r .access_token)

curl -s -X POST :3100/introspect \
  -H "Authorization: Bearer $SVC" \
  -d "token=$TOKEN" | jq .
# → { "active": true, "token_format": "jwt", ... }

# Revoke
curl -s -X POST :3100/token/revoke \
  -d "token=$TOKEN" → 204

# Verify revoked
curl ... → { "active": false }
```

## v1.0.0（保留 — User + OAuth2）

当前不做。等 v0.4.x 稳定后，v1.0.0 引入：
- User 模型 + 密码认证

- Role 管理
- Emerald entity_id 迁移

v1.0.0 代表 `/introspect` API 进入长期稳定——此后 breaking change 需要 v2.0.0。
